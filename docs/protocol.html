Comparison protocol

Quality, speed, memory, and mechanism are measured with distinct artifacts and joined only after provenance validation.

Questions and comparison roles

The primary question is whether VSA-PathMoE is non-inferior in conditional bits per UTF-8 byte to roneneldan/TinyStories-8M while improving CPU latency, generated-byte throughput, and resident memory under the same raw workload. TinyStories-33M is a predeclared larger-capacity anchor. The optional 1M/3M/8M/33M matrix describes scale; it cannot replace the primary baseline after results are known.

Shared raw text and separated splits

Training uses TinyStories-train.txt. The official validation file is split deterministically by a stable story hash: one half selects the custom checkpoint and the other is the held-out test set. The same held-out raw prefix and continuation are scored by every tokenizer. Conditional negative log likelihood is accumulated only over continuation tokens and divided by the continuation's UTF-8 byte count.

The raw boundary is moved to nearby whitespace so GPT-style leading-space pieces and SentencePiece whitespace pieces do not straddle the prefix/continuation boundary. Preparation fails if the custom tokenization of prefix + continuation differs from concatenating the separately tokenized pieces. This makes conditional bits per byte comparable despite different vocabularies.

Context policy

The custom decoder's native block is 256 tokens. The official TinyStories artifacts declare up to 2,048 positions, while the configured historical training support is 512. Primary quality uses each model's declared native comparison context: 256 for the custom checkpoint and 512 for official checkpoints. A separate 256-token common-context diagnostic isolates context-window effects.

Those two views are both retained. The report must never reinterpret an artifact's 2,048 positional capacity as evidence that it trained on 2,048-token sequences. The official revision, effective trained context, and raw artifact-config context are separate provenance fields.

A missing value is pending or unavailable, never a pass. The full verdict also requires artifact identity and workload compatibility. Dense controls are mandatory descriptive evidence and their artifacts, provenance, workloads, CPU environments, and parameter budgets are validated fail-closed. No preregistered matched-control performance threshold contributes to the core or final verdict.

Hardware separation and fairness

All custom and official quality artifacts for the official run are produced under the frozen DGX profile with CUDA BF16. CPU deployment artifacts are produced in independent processes on the same commodity x86 host, with CPU FP32, the same thread count, identical deterministic cases, equal warmup and repetition counts, KV caching, and 64 decode steps.

The custom runtime may use its documented fused shared-plus-expert kernel because this is the implemented deployment path. The report nevertheless publishes model bytes, router arrays, load time, cold and warm route-cache rows, and raw timing repetitions. Timing on the DGX Spark ARM host may diagnose that machine; it cannot satisfy the configured x86_64 ordinary-CPU claim.

Claim boundary. Official checkpoints are external baselines with different tokenizers, backbones, and training histories. They can support an executed system-level comparison, not a causal statement that VSA routing alone caused a difference.

Artifact join

The report accepts evaluation schema version 2 and benchmark schema version 3, selects the configured model list and primary ID exactly, recomputes workload hashes, verifies checkpoint and revision provenance, and requires the primary CPU device, dtype, architecture, and thread point. The generated comparison object carries each official model in frozen order; top-level verdicts are copied only from the primary 8M row.

When comparison.matched_control_profiles is configured, reporting also requires each named control's frozen config, training summary, custom evaluation, and custom CPU artifacts at the main run's exact thread set. It validates shared prepared semantics and runtime identity, then reports paired conditional-BPB uncertainty, HellaSwag delta, CPU bootstrap ratios, and measured/persistent-state memory ratios. These values are descriptive and do not create an undeclared pass threshold.
