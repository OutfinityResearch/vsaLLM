Operations and diagnostics

Operational commands are staged around immutable inputs and explicit preflight checks; recovery preserves the same experiment rather than mutating it.

Local engineering checks

npm run setup -- --locked-cpu
npm test
npm run smoke:offline
npm run docs:verify
npm run docs:serve-check

--locked-cpu installs torch==2.10.0 from the CPU index and the exact direct pins from requirements-dgx.txt. The offline smoke profile uses fixture data and tiny budgets to verify command plumbing. It does not download official checkpoints and cannot produce a scientific comparison. Use npm run doctor -- --profile <profile> to inspect resolved paths, runtime dependency versions, environment metadata, and parameter accounting before expensive work. Never run setup inside the NGC container.

./fileSizesCheck.sh is the project copy of the GAMP size and long-line checker. Its findings are maintainability signals, not model-quality tests.

Strict DGX preflight

npm run doctor:dgx -- --profile dgx_spark --strict exits unsuccessfully unless every implemented check passes. Run it inside the intended derived container, with the repository mounted and its identity variables exported.

Before preflight, use NVIDIA's supplied 240 W adapter, run the container with GPU access and --ipc=host, map the host UID/GID, and provide writable HOME and HF_HOME paths. These are manual operational preconditions; the current doctor records the environment but cannot identify the physical power adapter.

Command state machine

stateDiagram-v2
  [*] --> Doctor
  Doctor --> Download: preflight passes
  Download --> Prepare: manifest has immutable SHAs
  Prepare --> Train: format v5 validates
  Train --> Evaluate: best checkpoint v2
  Evaluate --> Transfer: archive CUDA/BF16 artifacts
  Transfer --> VerifyHandoff: schema v1 hashes + exact commit
  VerifyHandoff --> CPUBenchmark: x86 CPU/FP32
  CPUBenchmark --> Blind
  Blind --> Report
  Report --> [*]

Each transition has a persisted identity. Rerunning a completed preparation with changed semantics, resuming from a different environment, or reporting a different workload is rejected. Create a new profile for a changed experiment.

Create the transfer inventory with npm run handoff:create -- --profile dgx_spark after the sparse official evaluation and both controls' custom evaluations are complete. On the exact source commit at the x86 destination, run npm run handoff:verify -- --profile dgx_spark before any benchmark. Reinstall the CPU environment, run npm run download:deployment -- --profile dgx_spark, and never transfer .venv. The deployment-only manifest cannot prepare or train, and DGX preparation must never be rerun on x86.

Configuration loading also validates representable vocabulary and route widths, model geometry, positive batch/evaluation settings, nonnegative token budgets, learning-rate ordering, warmup and regularization bounds, generation indexes, unique benchmark targets, sufficient repetitions, context headroom, and the registered CPU/FP32/x86 primary runtime. Invalid long-run profiles stop before creating artifacts.

Common failures

Configuration drift

If the current merge differs from resolved_config.json, do not edit or delete the frozen file to make the command pass. Preserve the old run and select a new profile/run directory.

Prepared-data incompatibility

A mismatched signature, tokenizer SHA, router SHA, or routing mode means token IDs or routes may have changed meaning. Regenerate the store and retrain; do not patch checkpoint metadata.

Resume rejection

Use last.pt, not best.pt. Resume also rejects source, PyTorch, CUDA, container-digest, optimizer, or RNG changes. If the environment cannot be restored, begin a new run.

Missing official artifact

Recreate the exact configured model artifact at its immutable revision. Reporting never falls back from 8M to 33M and never accepts a reordered manifest.

Missing matched-control artifact

The main report fails closed when a configured control lacks its resolved config, training summary, custom evaluation, or any custom CPU benchmark in the main thread set. Run evaluate:custom on DGX and benchmark:custom:matrix on x86; do not duplicate official baselines or run a separate control report.

Wrong CPU runtime

Regenerate benchmark artifacts on the designated x86 host with CPU FP32. ARM timing, CUDA timing, or a filename copied from another thread count fails runtime validation.

Documentation partials do not load

Serve docs/ through HTTP. Browser security normally blocks fetch() for header, footer, and specification Markdown under a direct file: URL.

Operational stop conditions

Stop before training if the DGX doctor is not ready, the worktree is dirty, the NGC base lacks a recorded RepoDigest, direct dependencies are not exact, or immutable source snapshots cannot be resolved. Stop before benchmarking if the checkpoint/prepared provenance chain differs after transfer. Stop before reporting if any primary model, workload, runtime, or blind identity is missing.

Do not use npm run full for an official DGX profile. It conflates accelerator work with the designated x86 deployment stage and is explicitly disabled for dgx_spark.
