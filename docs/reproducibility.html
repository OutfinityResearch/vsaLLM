Documentation / Reproducibility

Identity across stages and hosts

Reproducibility contract

A result is reproducible only when configuration, source, data semantics, model artifacts, workloads, environments, and raw observations remain joinable.

Configuration freeze

Each command deep-merges config/base.json with the selected profile. The first artifact-producing command writes runs/<profile>/resolved_config.json. Later preparation, training, and evaluation commands compare the current resolved object with that file and reject drift.

Benchmark-only commands may vary device, dtype, and threads because the deployment host is intentionally different. The freeze comparison ignores only those runtime axes; the benchmark artifact itself records them, and reporting selects the configured CPU FP32 identity exactly. A changed model, source revision, workload, seed, threshold, context, or training budget requires a new profile and run directory.

Provenance chain

flowchart TD
  C[Resolved config SHA] --> P[Prepared signature v5]
  S[Dataset and model SHAs] --> P
  P --> K[Checkpoint v2 metadata]
  R[Runtime source SHA] --> K
  E[DGX environment + container digest] --> K
  K --> Q[Evaluation schema v2]
  K --> B[Benchmark schema v3]
  W[Workload manifest SHA] --> B
  Q --> O[Validated report]
  B --> O
  A[Blind evaluation ID] --> O

The runtime-source digest covers executable Python, Node scripts, configurations, dependency manifests, requirements-dgx.txt, .dockerignore, and the DGX Dockerfile. Git commit and dirty state are recorded separately. Environment snapshots also record installed direct-runtime dependencies, GPU name, compute capability, device memory, CUDA, cuDNN, and NVIDIA driver. Prepared semantics are linked by signature, tokenizer SHA, and router SHA. Checkpoint, evaluation, benchmark, and blind-review artifacts add their own identities rather than trusting filenames.

The DGX-to-x86 transfer adds handoff schema version 1. npm run handoff:create -- --profile dgx_spark hashes the prepared store and the sparse plus matched-control run directories only after required artifacts exist in a clean worktree. The x86 host must check out the manifest's exact commit and pass npm run handoff:verify -- --profile dgx_spark before benchmarking.

Mutable Hugging Face branches are resolved to immutable SHAs at download. The final report verifies the complete official list and selected revision. No stage may replace a missing identity with main, a nearby artifact, or the first matching file.

Determinism and uncertainty

The project seed drives preparation sampling, tokenizer/router construction, data splits, training route sampling, prompts, generation, and deterministic bootstrap seeds through explicit offsets. Recovery checkpoints preserve NumPy RNG and optimizer state. Cross-host transfer is safe at the semantic layer because prepared format 5 excludes local paths from its signature.

Determinism does not eliminate sampling uncertainty or hardware variability. The report preserves per-story quality rows and per-repetition timing rows, uses a paired story bootstrap for quality, and independently resamples process timing rows for speed ratios. Memory is a descriptive process observation without a confidence interval. Multiple training seeds remain desirable before a broad architectural claim.

Minimum archive

  • config/base.json and every executed profile
  • runs/<profile>/resolved_config.json
  • data/assets_dgx_spark.json and immutable snapshot identities
  • data/prepared/dgx_spark/PREPARED.json, tokenizer, router, and hashes
  • runs/<profile>/artifacts/ and both checkpoint files
  • runs/<profile>/evaluation/ and benchmark/ raw JSON
  • results/comparison_<profile>.json, reports, CSV, and frozen blind scores
  • handoff/dgx_spark_manifest.json and its verified transfer-root inventory
  • DGX doctor output, base RepoDigest, derived image ID, package freeze, driver/OS inventory
  • x86 doctor output, CPU model, affinity, governor, OS, and runtime inventory
  • Git commit, clean-state evidence, package manifest, and SHA-256 list of published artifacts

Keep the blind key private until judgments are frozen, then archive it separately with controlled release. Publish failed gates and incomplete criteria alongside successful ones.

The handoff deliberately excludes .venv, container filesystems, asset manifests, and model caches. Reinstall the destination with npm run setup -- --locked-cpu, which selects torch==2.10.0 CPU plus the exact direct pins, and regenerate assets from immutable revisions rather than transferring platform-bound state. Never run setup inside NGC.

Verification layers

npm test runs Node.js schema/report/orchestration tests and Python unit tests. npm run smoke:offline exercises a small local fixture without claiming scientific validity. Package checksum verification establishes distribution integrity. Documentation checks validate local links, partials, Mermaid inclusion, specs-loader targets, and the site over HTTP.

DS001 requires focused acceptance and rejection tests whenever configuration, preparation, routing, checkpoints, evaluation, benchmarks, reporting, or provenance change. A test that only accepts a good artifact is incomplete for a fail-closed boundary.

Publication wording

A completed report may support statements of the form “under the frozen TinyStories and commodity-x86 protocol, the named VSA-PathMoE checkpoint met or failed the registered gates against the pinned TinyStories-8M revision.” It does not support “VSA beats LLMs,” “new MoE proven,” or “7 MB memory” without the exact qualifying measurements.

Until all required artifacts exist, publish the design and protocol as a reproducible hypothesis. Do not convert absence of evidence into a zero, a pass, or a performance claim.
