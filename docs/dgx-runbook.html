DGX Spark runbook

Train and evaluate with CUDA BF16 on a clean DGX Spark container, then transfer the frozen experiment to a commodity x86 host for CPU FP32 measurement.

Stage 0 — freeze the run

Review the DS contracts, base and DGX profiles, registered gates, official revisions, prepared format, checkpoint cadence, and dense-control roles before inspecting final test outcomes. Commit the reviewed tree and begin from a clean worktree. Archive the Git commit and the package checksum manifest.

npm ci
npm test
npm run docs:verify
git status --short

The final line must be empty for the official run. Calibration belongs in dgx_spark_calibration and may establish batch safety or duration only; it must not change thresholds or select a favorable comparator after held-out results are known.

Stage 1 — pin and build the container

The Dockerfile default nvcr.io/nvidia/pytorch:25.11-py3 is a human-readable tag, not an immutable identity. Authenticate to NGC, pull the trusted tag, inspect its registry RepoDigest, and pass the full repository@sha256:… value as the Docker build base. Never invent or copy a digest from this documentation.

BASE_TAG=nvcr.io/nvidia/pytorch:25.11-py3
docker pull "$BASE_TAG"
BASE_REPODIGEST="$(docker image inspect \
  --format '{{index .RepoDigests 0}}' "$BASE_TAG")"
BASE_DIGEST="$(printf '%s' "$BASE_REPODIGEST" | sed 's/.*@//')"

docker build \
  --build-arg PYTORCH_IMAGE="$BASE_REPODIGEST" \
  -f containers/dgx-spark/Dockerfile \
  -t vsa-pathmoe:dgx-25.11 .

DERIVED_ID="$(docker image inspect \
  --format '{{.Id}}' vsa-pathmoe:dgx-25.11)"
printf '%s\n%s\n' "$BASE_REPODIGEST" "$DERIVED_ID"

requirements-dgx.txt pins direct experiment dependencies exactly and deliberately excludes PyTorch. PyTorch and CUDA come from the immutable NGC base and must not be replaced by pip. Archive the resolved base RepoDigest, derived image ID, Dockerfile, requirements lock, and build log.

Stage 2 — start the derived environment

Connect the supplied 240 W power adapter; NVIDIA documents it as required for optimal performance. Record the active system power and thermal configuration with the machine inventory. The cited DGX Spark guide does not prescribe a MAXN command, so do not import a Jetson-specific power command unless the exact deployed DGX OS documentation requires it.

mkdir -p "$HOME/.cache/huggingface"

docker run --rm -it --gpus all --ipc=host \
  --user "$(id -u):$(id -g)" \
  -e HOME=/tmp/vsa-home \
  -e HF_HOME=/cache/huggingface \
  -v "$PWD":/workspace/vsaLLM \
  -v "$HOME/.cache/huggingface:/cache/huggingface" \
  -e VSA_CONTAINER_IMAGE_DIGEST="$BASE_DIGEST" \
  -e VSA_DERIVED_IMAGE_ID="$DERIVED_ID" \
  vsa-pathmoe:dgx-25.11 bash

Host UID/GID prevents root-owned repository artifacts. --ipc=host supplies the shared-memory behavior expected by the framework container, while the explicit writable home and stable host cache keep downloads persistent without a root-owned named volume. The image already declares VSA_CONTAINER_IMAGE=nvcr.io/nvidia/pytorch:25.11-py3. Inside the container, record python -m pip freeze, Node, Python, PyTorch, CUDA, cuDNN, NVIDIA driver, GPU name, compute capability, device memory, DGX OS, and hardware inventory. Store evidence outside ephemeral container layers.

npm ci
npm test
npm run doctor:dgx -- --profile dgx_spark --strict

Do not run npm run setup inside NGC. The derived image already contains its NGC-provided PyTorch/CUDA stack and exact direct dependencies; setup could replace that controlled environment.

The strict doctor requires ARM64, NVIDIA GB10 with compute capability 12.1, CUDA and BF16, at least 110 GB reported system memory, at least 20 GB free repository storage, a clean worktree, a container runtime, the exact NGC tag identity, valid base and derived SHA-256 identities, exact direct dependencies, immutable hexadecimal commits for every configured dataset and official model, Node 22 or newer, and the CUDA/BF16 profile.

Before doctor execution, general config validation rejects unrepresentable token/route widths, invalid model geometry, empty or negative budgets, inconsistent learning rates, invalid evaluation/generation ranges, and benchmark settings that cannot produce the registered CPU/FP32/x86 comparison.

DGX preparation and training invoke the same doctor internally for every dgx_spark* profile. A missing manual preflight cannot make an invalid environment proceed.

Stage 3 — acquire and prepare once

npm run download:full -- --profile dgx_spark
npm run prepare -- --profile dgx_spark

Verify data/assets_dgx_spark.json contains the configured immutable revisions and data/prepared/dgx_spark/PREPARED.json reports format version 5. Archive its preparation signature, tokenizer SHA, router SHA, route distribution, source hashes, runtime-source identity, and preparation container/environment snapshot. The sparse and both dense profiles share this store.

Stage 4 — train and evaluate CUDA BF16

npm run train -- --profile dgx_spark
npm run evaluate -- --profile dgx_spark

npm run doctor:dgx -- --profile dgx_spark_dense_active --strict
npm run train -- --profile dgx_spark_dense_active
npm run evaluate:custom -- --profile dgx_spark_dense_active

npm run doctor:dgx -- --profile dgx_spark_dense_total --strict
npm run train -- --profile dgx_spark_dense_total
npm run evaluate:custom -- --profile dgx_spark_dense_total

All DGX profiles save format-version 2 recovery checkpoints every 2,000,000 target tokens. If interrupted, resume only from the profile's last.pt. Resume fails closed on resolved config, prepared identity, executable-source hash, PyTorch, CUDA, container digest, optimizer state, and RNG state.

After each profile, preserve resolved_config.json, artifacts/, checkpoints/, and evaluation/ on durable storage. When the sparse run, both controls, and all evaluations are complete, create the fail-closed handoff manifest:

npm run handoff:create -- --profile dgx_spark

The command requires a clean Git worktree, prepared format 5, the main and configured control run directories, training summaries, best and last checkpoints, custom evaluations, and every official evaluation in the primary run. It writes schema-v1 handoff/dgx_spark_manifest.json with the exact commit, transfer roots, byte sizes, and SHA-256 for every regular file, and prints the manifest SHA-256 for recording through an out-of-band channel; verification then enforces the exact profile, roots, complete file set, safe relative paths, sizes, and hashes. Do not run reports as a substitute for the missing CPU stage.

Transfer exactly the roots listed in the handoff manifest plus the manifest itself. They include data/prepared/dgx_spark and all three run directories. Do not transfer .venv, an interpreter environment, the DGX container filesystem, asset manifests, or Hugging Face caches.

On the x86 host, check out the manifest's exact Git commit, install the CPU environment locally, place the transferred roots at their repository-relative paths, and verify every byte before use:

npm ci
npm run setup -- --locked-cpu
npm run handoff:verify -- --profile dgx_spark
npm run download:deployment -- --profile dgx_spark

--locked-cpu force-installs the official torch==2.10.0 CPU wheel and every exact direct pin from requirements-dgx.txt, then checks the environment. Prepared format 5 is path-independent, while asset manifests contain local snapshot paths. The deployment download regenerates prompts and official snapshots from the same immutable configured revisions without the training corpus or HellaSwag. It cannot be used to prepare or train. Never run prepare for dgx_spark* on x86, edit revision fields, or modify prepared metadata by hand. Record CPU model, architecture, affinity, frequency governor, OS, Node, Python, PyTorch, all runtime dependency versions, and runtime-source hash.

Stage 6 — benchmark CPU FP32 on x86

Inspect lscpu -e=CPU,CORE,SOCKET,ONLINE, then select eight logical CPU IDs that map to eight distinct physical (socket,core) pairs without selected SMT siblings. Keep that same affinity for the sparse model, both official baselines, and both controls. Replace the placeholder only after inspecting the actual host; never copy an example core list blindly.

CPUSET=<eight-comma-separated-logical-CPU-ids>

taskset -c "$CPUSET" npm run benchmark:matrix -- --profile dgx_spark \
  --device cpu --dtype fp32

taskset -c "$CPUSET" npm run benchmark:custom:matrix -- --profile dgx_spark_dense_active \
  --device cpu --dtype fp32

taskset -c "$CPUSET" npm run benchmark:custom:matrix -- --profile dgx_spark_dense_total \
  --device cpu --dtype fp32

npm run report -- --profile dgx_spark
npm run blind -- --profile dgx_spark

The three DGX profiles freeze benchmark.required_cpu_affinity_logical_cpus=8 and benchmark.require_distinct_physical_cores=true. Before any custom or official adapter/model is loaded, the benchmark snapshots process affinity and Linux sysfs. It fails unless there are exactly eight unique affinity IDs, complete topology metadata, eight distinct (physical_package_id,core_id) pairs, and no selected SMT siblings.

The default matrix measures 1, 4, and 8 worker threads without changing the eight-CPU affinity. Each artifact's environment.cpu_topology.logical_cpus[] rows record logical_cpu, physical_package_id, core_id, and thread_siblings_list. Its summary records complete, affinity_logical_cpu_count, affinity_physical_core_count, and affinity_contains_smt_siblings.

Because the primary config declares matched controls, the main report fails closed until each control has its resolved config, training summary, custom evaluation, and the same CPU thread-set of custom benchmark files. The report revalidates every topology object and requires exact paired affinity, topology, and observed governors for sparse/official and sparse/control comparisons. It surfaces topology in JSON, Markdown, HTML, and CSV. The operator must keep one recorded governor policy stable, but the code intentionally imposes no particular governor value. Duplicate official evaluations and separate control reports are not required.

Review the generated blind form without the key, save the frozen scores, then rerun the primary report to join them. Preserve raw benchmark JSON, comparison JSON, Markdown/HTML reports, CSV, environment snapshots, and all SHA-256 values.

Stage 7 — release or reject the claim

Confirm that the report accepted the same custom checkpoint SHA and preparation signature across quality and CPU artifacts, the exact 8M and 33M revisions, the complete workload manifest, and the configured x86 CPU/FP32 runtime. Inspect every individual external-baseline gate and the main report's descriptive matched-control section. It reports paired conditional-BPB uncertainty, HellaSwag deltas, CPU speed uncertainty, and measured plus persistent-state memory ratios without inventing new control thresholds. Publish failures and uncertainty.

Never execute npm run full -- --profile dgx_spark. It is intentionally disabled. A monolithic run cannot represent the required DGX-to-x86 handoff.

Until stages 0–7 are complete and archived, the result remains pending. A calibration, smoke run, DGX-host ARM timing, or partial external-baseline evaluation is not evidence that VSA-PathMoE beats TinyStories.
