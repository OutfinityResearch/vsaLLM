Immutable sources, path-independent semantics

Data and preparation

Prepared format version 5 binds raw-data identity, tokenizer semantics, routing semantics, and deterministic splits without binding them to a workstation path.

Source resolution

The acquisition stage resolves TinyStories dataset and model revisions before downloading. The official DGX profile already names immutable revisions for the dataset, TinyStories-8M, TinyStories-33M, and HellaSwag. The resulting data/assets_dgx_spark.json records requested revisions, resolved SHAs, snapshot paths, and file identities. Evaluation and benchmarking validate the complete official-model list, order, ID, requested revision, resolved artifact, and trained context.

The scientific dataset roles are fixed: TinyStories-train.txt supplies tokenizer/router fitting and custom training; TinyStories-valid.txt supplies deterministic validation and test halves. Evaluation prompts.yaml is the generation prompt source. Official model snapshots are external baselines, not training inputs for the custom model.

npm run download:full -- --profile dgx_spark
npm run prepare -- --profile dgx_spark

Downloads and prepared data should reside on persistent storage mounted into the DGX container. Secrets and mutable caches are operational inputs; resolved manifest identities are scientific artifacts.

For every profile whose name starts with dgx_spark, prepare invokes the strict DGX doctor internally and stops if any preflight check fails. Running the doctor manually is useful evidence, but it cannot be bypassed by calling preparation directly.

Format version 5 pipeline

flowchart TD
  A[Resolved dataset manifest] --> B[Stream raw stories]
  B --> C[Reservoir samples]
  C --> D[Lossless SentencePiece BPE]
  C --> E[VSA router fit on prefixes]
  B --> F[Whitespace-safe prefix split]
  D --> G[Token arrays and offsets]
  E --> H[Route arrays and route indexes]
  F --> G
  F --> H
  G --> I[Prepared format v5]
  H --> I
  I --> J[Signature + tokenizer SHA + router SHA]

Preparation streams the multi-gigabyte source instead of loading it into memory. It takes deterministic reservoir samples for a 2,048-piece SentencePiece BPE tokenizer and for the router. Tokenization uses identity normalization, byte fallback, no dummy prefix, preserved whitespace, and explicit BOS/EOS IDs. A sample round-trip audit must reconstruct source text exactly; otherwise bits per byte would be ill-defined.

The request prefix is chosen near 96 characters at a stable whitespace boundary, and the continuation retains its separator. Only the prefix feeds the router. Each record stores BOS, prefix, continuation, EOS, its prefix-token length, route, and continuation target count. Training loss masks the conditioning prefix.

On-disk representation

  • data/prepared/dgx_spark/PREPARED.json — completion marker and full metadata
  • metadata.json — format, signatures, hashes, split and route statistics
  • tinystories_bpe.model — lossless custom tokenizer
  • router.npz — byte vectors, roles, and product prototypes
  • indexed/train/, validation/, test/ — uint16 tokens and NumPy indexes
  • eval/validation.jsonl, test.jsonl — raw prefix/continuation evaluation rows

Token streams use unsigned 16-bit values; offsets, prefix lengths, routes, target counts, route-sorted record order, and route boundaries are stored separately. This supports route-aware sampling without materializing all stories in memory. Per-split metadata records story counts, target tokens, active routes, route counts, and entropy.

Path-independent compatibility

The preparation signature hashes format version 5, project seed, semantic data configuration, dataset ID and resolved SHA, and the selected source filenames. It intentionally excludes local cache and mount paths. A prepared store may therefore move from the DGX host to an x86 benchmark host without changing token or route meaning. Metadata separately records the complete preparation environment, runtime-source identity, container tag, base digest, and derived image ID, so portability does not erase provenance.

A checkpoint embeds the prepared metadata. Loading, resuming, evaluating, and benchmarking fail closed unless the preparation signature, tokenizer model SHA, router artifact SHA, and routing mode are present and match. A legacy checkpoint without these fields is unverifiable and must be retrained.

--force is destructive. It removes and rebuilds the selected prepared directory. Do not rebuild data/prepared/dgx_spark while another stage reads it, and do not reuse checkpoints whose embedded preparation identity names the old store.

Sharing across sparse and dense profiles

dgx_spark, dgx_spark_dense_active, and dgx_spark_dense_total intentionally point to data/prepared/dgx_spark. Their source revisions and data configuration are identical, so one preparation supplies the same tokenizer, raw records, and split identities. Dense models map every stored route to route zero at model execution; they do not require a different text corpus.

Only the sparse profile should create the shared store in the official workflow. Freeze and archive PREPARED.json, metadata.json, tokenizer SHA, router SHA, and the asset manifest before any of the three training runs.

On the x86 deployment host, npm run download:deployment -- --profile dgx_spark regenerates prompts and official-model snapshots without the train corpus or HellaSwag. That manifest is intentionally insufficient for preparation. Never rerun prepare or train for a dgx_spark* profile on x86; verify the transferred prepared bytes through the handoff manifest instead.
