Training and checkpoints

The official sparse and dense-control runs use reviewed DGX Spark profiles, a shared prepared store, and format-version 2 checkpoints every two million target tokens.

Frozen training profiles

All three use device=cuda, dtype=bf16, four host threads, batch size 32, gradient accumulation 2, evaluation batch size 16, and data/prepared/dgx_spark. The shared base training budget is 320 million joint-phase target tokens followed by 80 million expert-specialization target tokens. The profiles alter only the architecture fields needed for their comparison role.

The container derives from the NGC PyTorch 25.11 image, but a mutable tag is not sufficient evidence. Before an official run, resolve the pulled base image to its registry RepoDigest, build and run from that immutable base, record the derived image ID and installed package freeze, and export the digest used by the fail-closed DGX doctor.

Training invokes that doctor internally for every dgx_spark* profile and refuses to begin unless it is fully ready. The explicit preflight command remains part of the run record; it is not the only enforcement point.

For DGX profiles, training also requires the prepared store's recorded runtime-source SHA, Git identity and clean state, PyTorch/CUDA/cuDNN/driver versions, GPU identity and compute capability, container tag, base digest, and derived image ID to equal the current training environment. Preparation, training, and resume therefore form one pinned accelerator provenance chain.

Two training phases

stateDiagram-v2
  [*] --> Joint
  Joint: all parameters trainable
  Joint: route sampling exponent 0.5
  Joint --> BestJoint: validation selects checkpoint
  BestJoint --> Experts
  Experts: only expert matrices trainable
  Experts: uniform over active routes
  Experts --> Complete
  Complete --> [*]

The joint phase trains embeddings, attention, shared feed-forward matrices, and every route expert. Routes are sampled from their training frequencies raised to exponent 0.5, reducing dominance by common routes without making the distribution fully uniform. Gradient accumulation is normalized by the number of valid continuation targets, so unequal microbatch padding does not distort the update.

At the phase boundary, specialization starts from the best joint validation checkpoint rather than the last update. Only expert input and output matrices remain trainable, and sampling is uniform over routes that occur in training. The same fixed validation batches select improvements. Dense controls have one model route and follow the same phase machinery.

Checkpoint contract

Checkpoint format version 2 stores model configuration and state, optimizer state where appropriate, complete training state, and metadata linking the profile, routing mode, resolved-config SHA, prepared metadata, environment, and parameter accounting. Writes use a temporary file followed by atomic replacement.

Every DGX profile checkpoints at each 2,000,000-target-token boundary and at phase completion. last.pt carries optimizer, NumPy RNG, gradient-scaler state, phase counters, global step, route usage, best loss, and the next evaluation and checkpoint boundaries. best.pt is selected on validation loss and deliberately omits optimizer state; it is for evaluation or phase initialization, not resume.

Resume and environment identity

A resume accepts only last.pt from the same resolved configuration and prepared semantics. It also compares executable-source identity, PyTorch version, CUDA version, and container digest, and requires optimizer and stochastic state. This prevents a hardware or software change from being silently folded into one training history.

npm run train -- --profile dgx_spark \
  --resume runs/dgx_spark/checkpoints/last.pt

If a compatibility field differs, create a new profile and run directory. Do not edit resolved_config.json or transplant optimizer state. Recovery is a continuation of one frozen experiment, not a mechanism for adapting the protocol.

Official execution order

npm run doctor:dgx -- --profile dgx_spark --strict
npm run train -- --profile dgx_spark
npm run evaluate -- --profile dgx_spark

npm run doctor:dgx -- --profile dgx_spark_dense_active --strict
npm run train -- --profile dgx_spark_dense_active
npm run evaluate:custom -- --profile dgx_spark_dense_active

npm run doctor:dgx -- --profile dgx_spark_dense_total --strict
npm run train -- --profile dgx_spark_dense_total
npm run evaluate:custom -- --profile dgx_spark_dense_total

Never run npm run full -- --profile dgx_spark. The monolithic command is disabled because GPU training/evaluation and commodity-x86 CPU benchmarking are different stages on different machines. Use the staged DGX runbook.
