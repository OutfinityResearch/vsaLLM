Symbolic route, tensor continuation

Architecture

A deterministic prefix router selects one small expert path while the decoder retains a shared attention and feed-forward backbone.

Request-level routing

The router consumes at most 512 UTF-8 bytes from the canonical conditioning prefix. It never sees the held-out continuation. Each byte has two independent bipolar vectors. The first 256-dimensional half bundles byte identity; the second binds byte identity to one of 64 periodic positional-role vectors before bundling. Ties at zero resolve to +1, producing a deterministic 512-dimensional bipolar code.

Two independently fitted binary k-means codebooks contain eight prototypes each. Int16 dot products rank the request code against each codebook; the Cartesian product of the two winning prototype indexes identifies one of 8 × 8 = 64 routes. Dot-product ranking is equivalent to Hamming ranking for bipolar vectors. The code notes that XNOR plus popcount could preserve route semantics, but this repository implements NumPy int16 products.

flowchart TD

Router fitting uses only training prefixes, a fixed seed offset, and 12 binary k-means iterations. Prepared metadata records route counts, entropy, prototype histories, router hash, and the causal-prefix invariant. This is a deterministic content-and-order router, not a learned token gate and not a claim that semantic relevance has already been proven.

Small tensor kernels

The decoder has a 2,048-piece SentencePiece vocabulary, width 192, six layers, six attention heads, learned positions, RMS normalization, tied token/output embeddings, and a 256-token block size. PyTorch scaled dot-product attention provides causal prefill and cached single-token decoding.

Each Transformer layer contains a shared feed-forward branch of width 128 and 64 route-specific micro-experts of width 57. Only the selected expert is used. In evaluation, its input and output matrices are concatenated with the shared branch and cached for up to four active routes. Two fused linear operations then compute both branches while preserving independent parameters. Moving the model between devices or dtypes clears this derived cache.

Why this is MoE-like, and what differs

Conventional sparsely gated MoE systems commonly select experts per token with a learned gate and may require balancing losses or distributed communication. VSA-PathMoE selects once per request outside the tensor decoder, activates the corresponding narrow expert at every layer, and reuses that choice for decoding. This reduces route work and makes the active tensor path predictable on a CPU.

The design is adjacent to deterministic hash routing and product-key access. Its specific combination—causal raw-prefix VSA encoding, two product codebooks, request-stable routing, and tiny per-layer experts—is the implementation under test. The documentation deliberately avoids a priority claim. Related work must be reviewed before describing it as a new MoE category; measured advantages must be restricted to executed baselines and hardware.

Controls that test the mechanism

dgx_spark_dense_active has one fixed route and exactly 1,755,840 parameters, matching the sparse model's arithmetic-active count. dgx_spark_dense_total also has one fixed route but expands the shared branch to width 3,719, producing exactly 10,029,504 parameters. Both retain the custom tokenizer, backbone family, preparation, and training protocol.

Within the sparse checkpoint, shared_only and permuted_route ablations test whether expert tensors and correct route identity reduce held-out loss. The dense profiles test whether an observed result follows merely from parameter budget or from conditional activation. Official Hugging Face checkpoints remain valuable external anchors, but their different tokenizer, backbone, and historical training protocol cannot isolate the routing mechanism.

Attribution standard. A fast external-baseline comparison is descriptive. A causal sparsity interpretation additionally needs separately trained dense controls, positive expert and routing ablations, frozen optimization budgets, and preferably multiple seeds.
