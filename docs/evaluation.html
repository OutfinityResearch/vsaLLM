Evaluation and reporting

Evaluation schema version 2 retains per-story quality, generated continuations, ablations, model identity, and environment evidence for each model.

Quality construction

Each adapter scores the same deterministic held-out rows from eval/test.jsonl. It receives the raw prefix and raw continuation, applies its own tokenizer, and accumulates negative log likelihood only for continuation tokens. Conditional bits per byte is NLL / (ln(2) × UTF-8 continuation bytes). Per-story NLL and bytes remain in the artifact so reporting can pair exact stories and bootstrap the aggregate ratio.

The custom adapter validates checkpoint format, prepared signature, tokenizer SHA, router SHA, routing mode, route count, and environment provenance before loading. Official adapters validate the selected manifest row and preserve model ID, immutable revision, effective 512-token training context, artifact-config context, parameter counts, and weight-file bytes.

Native and common contexts

The primary view is primary_quality_context=native: the custom model uses its 256-token block, while official checkpoints use the configured 512-token historical training support. A diagnostic reruns both at a common 256-token limit. Both results and their per-story rows are mandatory because they answer different questions.

The primary gate uses the upper 95% bound of a deterministic paired story bootstrap for the custom/official aggregate conditional-BPB ratio. A ratio at or below 1.02 is non-inferior under the registered margin. Point estimates alone do not pass the gate.

Secondary measurements

HellaSwag is a secondary diagnostic because TinyStories is a synthetic story domain, not a general commonsense corpus. hellaswag_examples=0 means the full configured validation split, not zero examples. The report retains raw and length-normalized accuracy and tests the normalized delta against −0.01.

The sparse custom evaluation also executes three fixed validation views: the full model, shared branch only, and a deterministic permuted route. The expert-contribution loss gain is shared-only − full; the routing-contribution gain is permuted-route − full. Each must reach 0.005 for the corresponding mechanism gate.

Generation uses the frozen prompt set, one greedy completion and three sampled completions per prompt, 128 new tokens, temperature 0.75, top-k 40, and top-p 0.95. These outputs are embedded in the full evaluation object and feed the blind-review packet.

Blind review without provenance fallback

npm run blind reads complete custom.json and primary official evaluation objects. It validates profile, model kind and ID, immutable official revision against the current manifest, custom checkpoint SHA, exact prompt and sample counts, aligned IDs, and byte-identical prompts. It does not fall back to detached generation files or another official checkpoint.

The exported form randomizes A/B position deterministically from a blindEvaluationId derived from both evaluation hashes and the prompt manifest. Keep results/blind_key_<profile>.json hidden. The evaluator opens only the form, exports scores with the same IDs, and freezes them before the key is joined during reporting.

Regenerating either model's continuations changes the evaluation hashes and requires a new blind review. A score file from an earlier packet is incompatible by construction.

Report validation and outputs

The report reads only the frozen resolved configuration and exact artifact filenames. It rejects missing schemas, reordered model manifests, mutable or mismatched revisions, incompatible checkpoint hashes, differing test rows, wrong native/common context modes, benchmark workload drift, wrong CPU runtime identity, dirty training provenance, and missing primary artifacts.

  • runs/<profile>/evaluation/custom.json
  • runs/<profile>/evaluation/official_roneneldan__TinyStories-8M.json
  • runs/<profile>/evaluation/official_roneneldan__TinyStories-33M.json
  • results/comparison_<profile>.json — machine-readable comparisons and gates
  • results/REPORT_<profile>.md and .html — human report
  • results/cpu_metrics_<profile>.csv — flattened CPU comparison rows

Every official model receives its own comparison row in configured order. Only TinyStories-8M populates the top-level core and final verdict. A missing 8M artifact stops reporting; TinyStories-33M is never promoted as a convenient replacement.

The main DGX report also resolves both configured dense-control profiles. Each must supply a frozen config, training summary, custom evaluation, and custom CPU matrix at the same thread values as the sparse run. Control artifacts are provenance-validated and summarized with paired quality uncertainty, HellaSwag delta, CPU speed uncertainty, and measured plus exact persistent-state memory ratios. They remain descriptive evidence rather than new post hoc gates.
