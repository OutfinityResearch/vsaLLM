Deployment evidence on commodity x86

CPU speed and memory

The primary efficiency claim is tested in separate CPU FP32 processes with a shared deterministic workload, explicit runtime identity, and measured resident memory.

Cross-host benchmark stage

The final checkpoint and CUDA/BF16 evaluation artifacts are produced on DGX Spark. The checkpoint, frozen resolved configuration, prepared format-v5 store, evaluation objects, and immutable asset identities are then transferred to a commodity x86_64 system. Official snapshots may be mirrored into a local cache, but their IDs and resolved revisions must remain identical.

Benchmark runtime fields are an explicit exception to training/evaluation freeze: device, dtype, and thread count may change for benchmark-only commands and are recorded in every artifact. All semantic fields remain frozen. First inspect the host topology, then select eight logical CPU IDs that represent eight distinct physical (socket,core) pairs without selected simultaneous-multithreading siblings. Every process must inherit the same affinity:

CPUSET=<eight-comma-separated-logical-CPU-ids>

taskset -c "$CPUSET" npm run benchmark:matrix -- --profile dgx_spark \
  --device cpu --dtype fp32

taskset -c "$CPUSET" npm run benchmark:custom:matrix -- \
  --profile dgx_spark_dense_active --device cpu --dtype fp32
taskset -c "$CPUSET" npm run benchmark:custom:matrix -- \
  --profile dgx_spark_dense_total --device cpu --dtype fp32

With no --threads override, orchestration executes 1, 4, and 8 worker threads. Those values do not resize the eight-logical-CPU process affinity. Custom and every configured official model run in independent Python processes, reducing load-order and allocator contamination. The custom-only control matrices do not rerun official baselines.

All three DGX profiles freeze benchmark.required_cpu_affinity_logical_cpus=8 and benchmark.require_distinct_physical_cores=true. The primary report requires the same thread set and exact affinity/topology/governor pairing for sparse, official, and dense-control artifacts. It also validates each control's resolved config, training summary, checkpoint provenance, preparation signature, workload, runtime source, and CPU environment before producing descriptive control ratios.

Deterministic multi-prompt workload

The benchmark derives four stable prompt samples at each 128, 320, and 512-character target from prepared held-out text. Each case carries its prompt hash, actual character and UTF-8 byte counts, and target. The canonical workload object contains 12 cases and a SHA-256 recomputed by the report for every model.

Each process performs five warmups and 30 measured repetitions, with 64 generated tokens and KV caching. Raw rows separate prompt prefill, decode, and end-to-end time and preserve generated token and UTF-8 byte counts. Both cold and warm route-cache cases are reported; the registered speed gates use warm-route rows because repeated deployment requests may reuse prepared fused route tensors.

flowchart LR
  W[12 hashed cases] --> C[Custom process]
  W --> O[Official process]
  C --> C1[5 warmups]
  O --> O1[5 warmups]
  C1 --> C2[30 raw rows × 64 tokens]
  O1 --> O2[30 raw rows × 64 tokens]
  C2 --> V[Manifest and runtime validation]
  O2 --> V
  V --> B[Independent bootstrap of medians]

Token throughput alone is not tokenizer-neutral. The protocol therefore reports end-to-end latency, decode tokens per second, and end-to-end generated UTF-8 bytes per second. A speed gate requires the minimum lower 95% bootstrap bound across prompt samples to reach 1.20×, preventing one favorable prompt from deciding the result.

Before adapter or model loading, each custom and official benchmark snapshots the current process affinity and Linux sysfs topology. Validation requires exactly eight unique affinity IDs, complete topology rows, eight distinct (physical_package_id,core_id) pairs, and no selected SMT siblings. An invalid host selection fails before model memory or initialization can contaminate a measurement.

Every artifact stores environment.cpu_topology. Each logical_cpus[] row contains logical_cpu, physical_package_id, core_id, and thread_siblings_list. The summary contains complete, affinity_logical_cpu_count, affinity_physical_core_count, and affinity_contains_smt_siblings.

The report independently revalidates that structure for every selected sparse, official, and control artifact, then requires exact paired affinity, topology, and observed-governor values at each worker-thread point. It surfaces topology in the comparison JSON, Markdown report, HTML report, and CPU CSV. The code does not prescribe a governor value; the operator fixes and records one stable policy, and report pairing prevents it from changing between compared processes. Four different memory quantities

“Active parameters” is an arithmetic count, not a residency measurement. The custom process loads the complete 10.03M parameter bank even though a request multiplies through only 1.76M parameters. The benchmark takes process snapshots before model load, after load, and after the workload, using RSS and, where the platform supplies them, USS and PSS.

QuantityMeaningUse

The measured component first uses a positive comparable USS load delta for both processes and otherwise a positive RSS load delta. Because this snapshot occurs before the first request, it excludes fused-route-cache population. A second component divides each model's exact persistent_model_state_max_bytes, which includes the custom router arrays and maximum four-route fused cache. The gate uses the larger available ratio, so allocator behavior cannot hide durable model state. The custom/8M value must be at most 0.80. Peak and total RSS remain visible diagnostics, while router and cache terms remain separately auditable.

Interpreting a result

A positive CPU result requires all prompt samples, correct runtime metadata, raw repetitions, confidence intervals, and a valid resident-memory source. It establishes a system-level result for this implementation, frozen workload, official revision, and executed CPU. It does not establish that every VSA router is faster, that the custom model uses only 7 MB of RAM, or that training on a GPU contributed to CPU efficiency.

DGX Spark has an ARM host. Timing there can be archived as a platform-specific diagnostic but cannot satisfy the configured ordinary-CPU gate, which explicitly requires commodity x86_64.
